# Автоматизация чтения тендерной документации: технический атлас

**Главный вывод:** автоматизировать извлечение данных из строительных тендеров РФ реально на 70–85%, но ни один формат не парсится «из коробки» — MVP должен строиться как **каскадный pipeline** с детерминированной базой (XML ЕИС + pdfplumber + openpyxl) и LLM-страховкой на сложных случаях. Ключевые узкие места — PDF-сканы со штампами, XLSX-ВОР с кастомной структурой каждого заказчика и отсутствие публичных парсеров формата `.gge`. Экономика: при грамотной архитектуре обработка одного тендера (~250 страниц) обходится в **4–170 ₽**, а минимально жизнеспособный стек собирается из бесплатных библиотек + Яндекс Vision OCR (~0,12 ₽/страница) + YandexGPT/GigaChat. BIM/IFC, DWG и TIMML для MVP нужно агрессивно де-приоритизировать: их реальная доля в тендерной документации на апрель 2026 остаётся ниже 10–20%, а трудозатраты несопоставимы с выигрышем.

Далее — подробный разбор по десяти блокам с конкретными библиотеками, ценами, ссылками на GitHub и рекомендуемой архитектурой.

## 1. ЕИС и XML-извещения: фундамент, у которого изменились правила игры

**С 01.01.2025 бесплатный FTP `ftp.zakupki.gov.ru` закрыт**, а с 04.10.2025 вся интеграция переведена на ГОСТ-шифрование через домены `int.zakupki.gov.ru` и `int44-ttls-cert.zakupki.gov.ru` (новость ЕИС № 34888, подтверждение CNews от 10.01.2025). Массовая выгрузка «слепка» базы больше невозможна — каждый запрос авторизуется токеном через Госуслуги и логируется. **Для MVP это критично:** Telegram-бот на обычном VPS вынужден работать либо через публичный SOAP-сервис `getDocsIP` с токеном физлица (лимит ~5000 запросов/час), либо через агрегатор — Clearspending, Spending.gov.ru, DaMIA, intec-balance (последний предоставляет коммерческое зеркало FTP).

**XSD-схемы ЕИС** публикуются в «Альбоме ТФФ» (текущая версия 13.x) на `zakupki.gov.ru/epz/main/public/document/view.html?sectionId=432`. Схемы меняются мажорно раз в 6–12 месяцев, итерации — ежемесячно; в корне каждого XML обязателен атрибут `schemeVersion`, поэтому в парсере нужно закладывать логику выбора маппинга по версии. Хабр-статья 869934 (декабрь 2024) — самая актуальная реализация SOAP-клиента на Python с комментариями по миграции 2025 года.

**Ключевые поля, доступные программно** в XML извещений (`epNotificationEF2020` и родственные типы): `purchaseNumber`, `lot/maxPrice` (НМЦК), `lot/purchaseObjects/purchaseObject` с ОКПД2/КТРУ и структурированными характеристиками (обязательны с 01.10.2023), обеспечение заявки и контракта, `procedureInfo` со сроками, `placeOfDelivery` с ФИАС/ОКТМО, `attachmentsInfo/attachment` с URL приложений, а с 01.04.2024 — структурированный `contractProject`. **Однако для строительных тендеров основная масса данных — объёмы работ, спецификации, расценки — по-прежнему живёт в приложенных PDF/XLSX/GGE-файлах**, а не в структурированных полях извещения. Структурированные характеристики работ заполняются заказчиками формально и часто противоречат ТЗ (Решение Псковского УФАС 060/06/48-130/2024).

**Python-стек:** `requests` + `lxml` (XPath с `local-name()` для устойчивости к смене namespace) + `xmltodict` + `pydantic` для типизированных моделей + `zeep` при желании использовать полный SOAP-клиент. Open-source проектов с ЕИС на GitHub много, но все < 10 звёзд и устарели: самые полезные — **aksyuk/zakupki_gov_ru** (CSV с XPath-шаблонами для разных типов XML, НИУ ВШЭ) и **Homyakin/ZakupkiParser** (Java, наиболее полный обход FTP-структуры). Коммерческие мониторинги (Seldon, Tenderplan, Контур.Закупки) построены по одному паттерну «сырой XML → XSD-парсинг → Elastic/Postgres → API»; их ключевая функциональность для стройки — **скачивание приложений и OCR-индексирование содержимого**, что и есть основная задача описываемого продукта.

**Рекомендация для MVP:** первую итерацию построить на **Clearspending REST API** (`openapi.clearspending.ru/restapi/v3/`, JSON, бесплатно, задержка 1–2 дня) + DaMIA для детализации, фильтруя по ОКПД2 групп 41.xx/42.xx/43.xx/71.12.xx. Переход на собственный SOAP-клиент с токеном — вторая итерация, когда появятся требования к оперативности ≤1 часа.

## 2. Формат .gge — открытая спецификация, практически отсутствующая экосистема

`.gge` — это обычный XML по XSD-схемам Минстроя, переименованный для идентификации на платформе Главгосэкспертизы. Схемы публикуются на **minstroyrf.gov.ru/tim/xml-skhemy/**, валидируются через **checkxml.gge.ru** и КПСР. Ключевая хронология: ЛСР версии 1.09 с 18.08.2021, обязательный XML для экспертизы с 01.08.2023, **XML-схема ВОР `QuantityTakeoff-3_01.xsd` обязательна с 11.01.2025**. Действующие схемы покрывают ЛСР (БИМ и РИМ), объектную смету, сводный сметный расчёт, конъюнктурный анализ (КАЦ), ВОР и пояснительную записку.

Внутри `.gge` лежит полная структура сметы: шапка с привязкой к ФРСН и ФСНБ-2022, разделы (без подразделов), сметные строки с обоснованием (ФЕР/ТЕР/ГЭСН/ФССЦ/ФСЭМ), объёмы, прямые затраты (ОЗП/ЭМ/ЗПМ/МАТ), коэффициенты (зимнее удорожание, стеснённость), нормативы НР и СП, индексы перевода цен. **НДС и лимитированные затраты в ЛСР не попадают** — они выносятся в ССР и ОС. Круговая конвертация `.gsfx → .gge → .gsfx` возможна, но теряет коэффициенты и макросы конкретной программы.

**Главная экосистемная проблема:** публичных open-source парсеров `.gge` **практически не существует**. Поиск по GitHub по ключам «gge parser», «грандсмета», «смета XML», «Главгосэкспертиза XML» не даёт ни одного зрелого проекта. Коммерческий онлайн-конвертер есть у cesnnov.ru (только РИМ). Поддерживающие программы (ГРАНД-Смета, Smeta.RU, 1С:Смета, АРОС-Лидер, Турбо Сметчик, Комплекс А0) экспортируют единообразный XML — это требование КПСР, которая играет роль де-факто арбитра формата — но **межпрограммный обмен всё ещё теряет данные** в коэффициентах и оформлении.

**Практический путь:** скачать XSD с Минстроя, сгенерировать привязки через `xmlschema` или `generateDS`, написать собственный парсер с учётом версий. `xmlschema.to_dict()` даёт готовый словарь с валидацией за 10 строк кода. Из `.gge` восстанавливается **полная структура сметы** — в отличие от PDF/XLSX, это детерминированный источник. Если заказчик прикладывает `.gge`, это **самый ценный файл в тендере** для автогенерации прототипа сметы.

## 3. XLSX ВОР — хаос форматов при наличии стандартов

До **11.01.2025** единой формы ВОР не существовало: ориентиром служили ГОСТ 21.111-84 (рекомендательный), письмо Минстроя 31983-ОГ/03 и Методика 421/пр п.27,35. С 11.01.2025 для госэкспертизы ВОР обязательна в XML-формате, но **в тендерных пакетах 44-ФЗ/223-ФЗ Excel остаётся де-факто нормой** — и будет оставаться ещё годы. Типовая структура колонок: № п/п, обоснование/шифр, наименование работ, единица измерения, объём, формула расчёта, наименование файла ПД, ссылка на чертёж, примечание. Но каждый заказчик добавляет свои вариации: КС-2, КС-3, КС-6а, М-29, дефектные ведомости, сопоставительные ВОР, региональные ТСН-формы.

**Типичные технические сложности:** объединённые ячейки в многоуровневых шапках («в т.ч. ОЗП/ЭМ/МАТ»), скрытые строки со служебными формулами, формулы с внешними ссылками (`=[Book2.xlsx]Sheet1!$A$1`) без сохранённых значений, логотипы-изображения в шапке, несколько логических таблиц на одном листе, смешанные типы в ячейке («12,5 м²»), числа как текст с неразрывным пробелом `\xa0`, защищённые листы, старые `.xls` в BIFF, разные десятичные разделители.

**Сравнение Python-библиотек** на файле 500k строк (бенчмарк Haki Benita, январь 2024): **python-calamine — 3,58 с** (Rust-ядро, победитель по скорости), polars/DuckDB ~4 с, pandas+openpyxl ~33 с, openpyxl напрямую ~35 с. Рекомендуемая связка: **openpyxl** для понимания структуры (доступ к `merged_cells`, видимости строк, стилям, формулам с `data_only=True`) + **python-calamine** для массовой обработки + **pandas/polars** для хранения в parquet.

**Маппинг колонок** — центральная задача ВОР. Гибридная схема даёт лучший результат: (1) нормализация заголовков → (2) `rapidfuzz.process.extractOne` с `token_sort_ratio` против словаря канонических вариантов («наименование работ», «описание», «работы и затраты», «виды работ») — score ≥ 70 обычно достаточно → (3) детекция строки-шапки через подсчёт совпадений ≥ 3 колонок → (4) для нестандартных файлов fallback на LLM со structured output (Pydantic schema, первые 40 строк как CSV). Производственные данные подсказывают: чистая эвристика даёт 75–85% точности маппинга, эвристика+rapidfuzz — 85–92%, fastText-классификатор на 10k размеченных заголовков — 93–96%, гибрид с LLM fallback — **96–99%**. На задачу распознавания структуры Excel-таблиц Microsoft Research в SpreadsheetLLM (arXiv 2407.09025) показала F1=0,759 для GPT-4 с сокращением токенов на 96% через structural anchoring.

**Типичные ошибки при автоматическом извлечении:** смешение «количество» и «формула» в одной ячейке (нужен regex-экстрактор числа), дубли единиц измерения (м², m2, кв.м), кратность измерителя (расценка «1000 м³» при ВОР в «1 м³»), неразрывные пробелы, путаница промежуточных итогов со строками работ, «дубль-шапки» при переносе страниц.

## 4. PDF — основной объём работы и главная технологическая развилка

### 4.1. Цифровые PDF: связка PyMuPDF + pdfplumber остаётся SOTA

Для нативных PDF, экспортированных из MS Word или «ГРАНД-Сметы», достаточно бесплатных инструментов. **PyMuPDF (fitz) 1.26+** лидирует по скорости (~0,1 с/страница), даёт отличную кириллицу и структуру через `page.get_text("dict")`. **pdfplumber 0.11** — лучший по работе с таблицами в rule-based категории (в бенчмарке arXiv 2410.09871 на категории «Tender» из DocLayNet Camelot показал recall 0,72). **pymupdf4llm** — ключевая находка 2024–2025: превращает PDF в чистый Markdown с заголовками, готовый для LLM/RAG. **pdfminer.six** в 25 раз медленнее PyMuPDF и используется только как fundament для pdfplumber. **pypdfium2** — самый быстрый (~0,003 с/страница), но без структурного анализа. Рекомендация для MVP: pymupdf4llm как первый проход + pdfplumber для таблиц + детектор сканированных страниц (`len(page.get_text()) < 100 and page.get_images()`).

### 4.2. OCR для русского: Яндекс Vision побеждает по соотношению цена/качество

**Tesseract 5** с `rus.traineddata` из `tessdata_best` — рабочий бесплатный baseline, CER 2–5% на чистом 300-dpi скане, но деградирует до 10–25% на искажённых сканах, штампах и шрифтах < 9 pt. **PaddleOCR PP-OCRv5/VL 1.5** (январь 2026) — текущий лидер open-source на кириллице, PaddleOCR-VL 1.5 взяла первое место в OmniDocBench v1.5 (92,86%), Apache-2, умеет таблицы → HTML. **Surya** (datalab-to/surya) поддерживает 90+ языков, но имеет нестандартную лицензию AI Pubs Open RAIL-M с порогом выручки. **EasyOCR** медленный на CPU, **docTR** (Mindee) слабо поддерживает кириллицу.

Среди коммерческих API расклад для русского рынка однозначен: **Яндекс Vision OCR** — **≈ 0,12 ₽/страница** (120 ₽ без НДС за 1000 единиц), данные остаются в РФ по УЗ-1 152-ФЗ, хорошо распознаёт штампы. Google Cloud Vision OCR — $1,50/1000 страниц, но слабее на печатях. **Azure AI Document Intelligence prebuilt-layout** — $10/1000 (commitment до $0,53/1000 при 1M+ страниц) — **сильнейший западный вариант по русским таблицам и merged cells**. Amazon Textract официально не поддерживает русский и путает латиницу с кириллицей в шифрах ФЕР. **ABBYY FineReader Server** — исторический референс качества (~1,6–8 ₽/страница), оптимум для регуляторно-чувствительных проектов on-prem. **Mistral OCR 3** — дешёвый ($1–2/1000 в Batch), но SaaS за границей.

**Новые инструменты 2024–2026** заслуживают отдельного внимания. **Docling** (IBM, MIT, github.com/docling-project/docling — ~53,7k звёзд) — модульный pipeline с DocLayNet-layout + TableFormer, ~0,49 с/страница на L4 GPU, даёт структурированный JSON и Markdown. **Marker** (datalab.to/VikParuchuri) использует Surya под капотом, olmOCR-Bench 76,1, GPL-лицензия. **MinerU 2.5** (OpenDataLab, AGPL) — ~0,21 с/страница, SOTA-качество. **olmOCR 2** (AllenAI, октябрь 2025) — Qwen2.5-VL-7B fine-tune с GRPO, лидер olmOCR-Bench (82,4), но оптимизирован под английский. **Nougat** (Meta) — узкоспециальный для scientific PDF, не обновляется. **LlamaParse** — SaaS, $1,25–90 за 1000 страниц в зависимости от режима.

**Предобработка обязательна** для низкокачественных сканов и даёт +30–60% к CER любого движка: upscale для мелкого шрифта (`cv2.resize` с `INTER_CUBIC`, ×2), deskew через `cv2.minAreaRect`, `fastNlMeansDenoising` для JPEG-артефактов, **CLAHE критичен для штампов и печатей** (неравномерный контраст), бинаризация Sauvola (skimage) предпочтительнее Otsu для документов с неравномерным освещением.

**Экономика при 100 тендеров/день ≈ 450k страниц/месяц:** Tesseract на VPS — 3–5 тыс. ₽/мес посредственного качества, PaddleOCR-VL на одной RTX 4090 — 25–40 тыс. ₽/мес SOTA-качества, Яндекс Vision — ~54 тыс. ₽/мес без капекса и с 152-ФЗ-комплаенсом, ABBYY Vantage on-prem — 500 тыс.–3,6 млн ₽/мес. **Рекомендуемый гибрид**: pymupdf4llm для цифровых + PaddleOCR-VL локально + Яндекс Vision для сложных случаев. Капекс 250–500 тыс. ₽ на GPU окупает себя за 4–6 месяцев при устойчивом потоке.

### 4.3. Таблицы в PDF: Camelot lattice + Яндекс Vision покрывают 95% случаев

Для нативных PDF-смет с рамками **Camelot lattice** (требует Ghostscript) даёт 88–94% cell-accuracy на локальных сметах из «ГРАНД-Сметы». **pdfplumber** с настройкой `extract_tables` — лучший визуальный контроль через `debug_tablefinder`. **Tabula-py** уступает Camelot. **img2table** — оптимальный open-source для сканов, поддерживает `lang="rus"`, `detect_rotation=True` и `borderless_tables=True` (alpha). **Table Transformer (TATR)** от Microsoft на PubTables-1M — 81% exact match на ICDAR-2013 combined; хорош для объединённых ячеек (распознаёт класс `spanning cell`), но требует GPU.

**Облачные сервисы** на сканах сметной документации: **Azure prebuilt-layout $10/1000 — лидер по русским таблицам** (88–94% accuracy, явные `RowSpan`/`ColumnSpan`), **Яндекс Vision с моделью `table`** — единственный вариант с 152-ФЗ-комплаенсом (88–94% accuracy). Amazon Textract Tables — $15/1000, страдает на коротких числовых шифрах ФЕР (путает С/C, Е/E). Google Document AI Layout Parser — $10/1000 flat, Gemini-based.

| Тип документа | Camelot | pdfplumber | img2table+rus | TATR | Textract | Azure layout | Яндекс table |
|---|---|---|---|---|---|---|---|
| Локальная смета, нативный PDF | 88–94% | 82–90% | 70–80% | 65–78% | 80–88% | **92–97%** | 90–95% |
| ВОР по форме ГГЭ | 85–92% | 80–88% | 65–78% | 60–72% | 78–85% | **90–95%** | 88–93% |
| Скан сметы 300 dpi | 0% | 0% | 70–82% | 55–72% | 85–92% | 88–94% | **88–94%** |
| Фото 150 dpi | 0% | 0% | 45–65% | 30–55% | 70–82% | 75–85% | **80–90%** |

Ключевые лайфхаки: шифры ФЕР нормализовать через whitelist `[А-Яа-я0-9\-.]`, тире унифицировать, при Camelot использовать `copy_text=['h','v']` для merged cells и хранить две версии DataFrame (raw + denormalized), многостраничные таблицы склеивать по совпадению шапки. **Самый большой выигрыш даёт не выбор инструмента, а доменная пост-обработка**: валидация сумм строк = итог раздела ± 1 ₽, справочник-матчинг шифров ФЕР/ГЭСН из ФСНБ-2022, template-based проверка колонок ВОР. Это поднимает сквозную точность с 70–80% до 95–98%.

### 4.4. LLM-слой: структурированное извлечение и каскадный роутинг

**Цены флагманов на апрель 2026** (USD input/output за 1M токенов): GPT-4.1 — $2/$8 с контекстом 1M; Claude Sonnet 4.6 — $3/$15, long-context премия отменена в феврале 2026; Gemini 2.5 Pro — $1,25–2,50/$10–15; GPT-4o-mini — $0,15/$0,60; Gemini 2.5 Flash-Lite — $0,10/$0,40; DeepSeek V3.2 — $0,28/$0,42 (но данные уходят в КНР). **Российские**: YandexGPT 5 Pro — 400 ₽/1M (снижено втрое в августе 2025), GigaChat 2 Max — 650 ₽/1M pay-as-you-go, GigaChat 2 Lite — 65 ₽/1M, **T-Pro 2.1 Т-Банка — Apache 2.0, бесплатно на self-host** (Qwen3-32B с русским дообучением) — лучший open-weight для on-prem.

**Экономика одного тендера (250 страниц, ~250k input tokens, 10k output):** GPT-4o-mini text — 4 ₽, Gemini 2.5 Flash — 9 ₽, DeepSeek V3.2 — 7 ₽, GPT-4.1 text — 52 ₽, Claude Sonnet 4.6 text — 81 ₽, Gemini 2.5 Pro text — 37 ₽, YandexGPT 5 Pro — 104 ₽, GigaChat 2 Max — 169 ₽. Multimodal vision в 2–5 раз дороже: GPT-4o vision — 98 ₽, Claude Sonnet vision — 115 ₽, Gemini 2.5 Flash vision — 13 ₽. **На 1000 тендеров/мес реалистичный бюджет — 10–170 тыс. ₽** в зависимости от стека. Prompt caching даёт −50–90% на повторяющихся префиксах, Batch API — −50%, оба паттерна обязательны к внедрению с первого дня.

**Pipeline-паттерны** (рекомендуется hybrid каскад):
1. pdfplumber/PyMuPDF на страницу → если извлекается текст и таблица консистентна, идём в текстовый LLM.
2. Иначе рендер в PNG (300 dpi) → vision-LLM бюджетного класса (Gemini 2.5 Flash, Claude Haiku 4.5) для точечной разметки.
3. Критичные поля (НМЦК, сроки, итоги смет) — повторный запрос на флагман (GPT-4.1/Claude Sonnet 4.6) с CoT и source grounding.
По прод-данным, 85% страниц закрывается бюджетной моделью, экономия 60–80% против «всё через флагман».

**Structured outputs — обязательный инструмент.** `instructor.from_openai(client, mode=instructor.Mode.TOOLS)` с Pydantic-схемой `TenderSummary` (поля `nmck_rub: Decimal`, `vat_included: bool`, `work_items: list[WorkItem]`, `extraction_confidence: float`) плюс `max_retries=3` даёт 95%+ валидных извлечений. Альтернативы: `outlines`/`guidance`/`llguidance` для constrained decoding на локальных моделях.

**Галлюцинации на числах — главный риск.** LLM токенизируют «15 478 312,50» как 8–11 токенов с неразрывными пробелами, у моделей запятая vs точка как десятичный разделитель срабатывает непредсказуемо. Защита:
- **Source grounding:** в каждое числовое поле зашивать `source_page`, `source_bbox`, `source_quote` (80–120 символов вокруг числа). Без цитаты поле помечается как не извлечённое.
- **Regex post-checks** против исходного текста: `\d[\d\s]*[,.]?\d{0,2}\s*(руб|рублей|₽)`.
- **Cross-model verification:** два прохода разными семействами (GPT-4.1 + Claude или YandexGPT Pro + GigaChat Max), majority vote, расхождение > 0,5% → эскалация к o3 или Opus 4.6 с CoT.
- **Доменные правила:** обеспечение заявки ≤ 5% НМЦК при > 1 млн; сумма позиций сметы ±0,5% = НМЦК; сроки ≥ минимальных по ст. 34 44-ФЗ.
- **Confidence scoring:** порог 0,85 для авто-загрузки, ниже — ручная модерация.

**LLM нельзя доверять безусловно** на: итоговых суммах смет с округлением ниже копейки (всегда детерминированный пересчёт), таблицах с мульти-row/colspan без видимых границ (10–20% ошибок), сканах 150 dpi и ниже, юридических порогах (антидемпинг, ПП 2571/1875) — это отдельный rules engine.

## 5. DWG — отложить в пост-MVP, кроме извлечения штампа

DWG встречается в 30–50% тендеров как дополнение к PDF, почти никогда как единственный источник. **Ценность для сметы ограничена**: экспликации помещений с площадями (30–60% случаев), спецификации оборудования (10–20%), штамп с метаданными проекта (почти всегда). Проблема в том, что в российской практике таблицы часто «рисуются» линиями + MTEXT вместо использования класса `ACAD_TABLE`, что делает извлечение без визуального анализа крайне нестабильным.

**Инструментарий:** прямое чтение DWG без AutoCAD в open-source закрыто — **ODA SDK (Teigha)** стоит **$7,500 первый год + $4,500/год**, **LibreDWG** на GPLv3 блокирует коммерческое использование в закрытом продукте, **ezdxf 1.4.3** (MIT, mozman/ezdxf) читает только DXF. Практический путь: `ODA File Converter` (бесплатный CLI) → DXF → `ezdxf.recover.readfile()` с аудитом повреждений → извлечение MTEXT/TEXT, блоков INSERT со штампами, ACAD_TABLE (если есть) и замкнутых LWPOLYLINE для помещений (через `shapely.Polygon`). Альтернатива — `DWG → PDF` через тот же ODA File Converter и прогон через PDF-pipeline.

**Рекомендация для MVP:** **не браться за геометрический расчёт площадей** — это research-задача. Ограничиться (1) конвертацией DWG→PDF и использованием существующего OCR-pipeline, (2) извлечением блока штампа из DXF через ezdxf — это почти всегда работает и даёт метаданные проекта дёшево. **Не покупать ODA SDK** на стадии MVP, **не использовать LibreDWG** в закрытом продукте, **не полагаться на Autodesk APS** из РФ из-за санкционных рисков.

## 6. BIM/IFC — опциональная ветка через IfcOpenShell, TIMML отложить

Несмотря на нормативные требования (ПП 331 от 05.03.2021 с редакцией ПП 2357, ПП 614 от 17.05.2024 — с обязательным ТИМ для застройщиков 214-ФЗ по МКД с 01.07.2024), **реальная доля тендеров с IFC-вложением на апрель 2026 — 10–20%**. По данным ДОМ.РФ, к октябрю 2024 г. ТИМ применяли ~26% застройщиков, к началу 2026 — оценочно 30–44%, но подавляющее большинство использует ТИМ внутренне, а подрядчики получают PDF рабочки. В госэкспертизу ЦИМ подаётся в IFC 4.0+, но это касается не тендеров, а экспертизы.

**IFC содержит идеально структурированные данные для сметы:** `IfcElementQuantity` с `Qto_WallBaseQuantities.NetVolume/NetSideArea/Length` — это **готовая ведомость объёмов**, `IfcMaterialLayerSet` — спецификация материалов по слоям, `IfcClassificationReference` — привязка к КСИ/Uniclass. Основной инструмент — **IfcOpenShell** (LGPL-3.0, ifcopenshell.org, github.com/IfcOpenShell/IfcOpenShell) с Python API, поддержкой IFC2x3/IFC4/IFC4x3, утилитами ifcpatch/ifctester/ifcdiff/ifcclash. Если `Qto_*` в модели отсутствуют (Revit по умолчанию их не экспортирует), запускается `IfcConvert --calculate-quantities` для обогащения геометрически. Альтернативы: **xBIM Toolkit** (CDDL, permissive, для .NET), **IFC.js/web-ifc** (MIT, WASM в браузере).

**RVT (Revit) парсить не пытаться.** Формат закрытый, ODA BIM SDK стоит ~$13,75k первый год, APS недоступен из РФ. Стандартный workflow — требовать IFC-экспорт от проектировщика.

**TIMML 1.02** в исходной задаче приписан НОТИМ — это **неточность**. Разработчик — **АРСИТИМ** (arsitim.ru/timml), ассоциация сметного ПО (АВС-Н, Визардсофт, Гектор, Эрикос-ЦСП). TIMML — не аналог IFC, а **формат передачи готовой сметы с обратными ссылками на GUID элементов IFC-модели** для автообновления. Утверждение, что «TIMML закреплён в ПП 614» — интерпретация отраслевых СМИ; в самом ПП 614 TIMML не упомянут. Публичных парсеров на GitHub нет, но поскольку это XML с открытой спецификацией — читается любым XML-парсером. **Распространённость на апрель 2026 — околонулевая.**

**Рекомендация для MVP:** (1) реализовать опциональную ветку «если в тендере есть `.ifc` — IfcOpenShell извлекает Qto + материалы + классификацию», трудозатраты 2–4 недели middle-инженера; (2) TIMML — только мониторить; (3) RVT — отклонять или требовать IFC; (4) заложить плагинный слой для форматов, чтобы позже подключать новые парсеры без переписывания.

## 7. Архивы: кириллические имена — главная боль, всё остальное решено

**ZIP-кодировки** — основная практическая проблема. Файлы, упакованные на российском Windows WinRAR/7-Zip без UTF-8 flag (bit 11), приходят в **CP866 или CP1251**, а стандартный `zipfile` декодирует их через CP437 → получается мохобёр. Стандартный трюк: `filename.encode('cp437').decode('cp866')` — широко цитируется, но остаётся эвристикой. **Более чистое решение — `libarchive-c`** с аргументом `header_codec='cp866'`, плюс дефолтные флаги безопасности `SECURE_NODOTDOT`/`SECURE_NOABSOLUTEPATHS`/`SECURE_SYMLINKS` с версии 3.0.

**Стек библиотек:** `zipfile` (stdlib) для ZIP, **`rarfile` 4.2** (ISC) для RAR3/RAR5 с внешним `unrar`/`bsdtar`, **`py7zr`** (LGPL, miurahr/py7zr, ≥ 0.20.1 из-за CVE-2022-44900) для 7z с 7zAES, **`patool`** как универсальный wrapper для 40+ форматов. Для детекции формата — `python-magic` поверх libmagic вместо расширений.

**Защитные меры обязательны** в production: max depth рекурсивной распаковки (тендерные архивы часто содержат ZIP внутри ZIP с подписанными `.sig` файлами), cumulative decompression ratio limit, max_files, max_total_size. Библиотеки: `safezip`, `sunzip`, `unzipbomb`. **Запароленные архивы в тендерах встречаются редко** — это противоречит принципу публичности 44-ФЗ; детектить через `is_encrypted` и отправлять на ручную обработку.

**Классификация содержимого архива** по именам файлов через regex на русские ключевые слова («смета», «тз», «техническое задание», «проект контракта», «ведомость», «чертёж», «ВОР», «ЛСР», «ПСД») работает на 70–85%, остальные — через MIME + содержимое (ML-классификатор типа fastText на 5–10k размеченных имён или LLM для сложных случаев).

## 8. Защита документов на ЭТП — почти не препятствие

**Для ЕИС** документация публична по 44-ФЗ и доступна без авторизации. На крупных ЭТП (Сбер-АСТ, РТС-тендер, ЕЭТП, ТЭК-Торг) часть документации требует регистрации, но сами файлы после скачивания обычно не защищены. **Encrypted PDF (с паролем) в тендерах — редкость**; для открытия без пароля применяются `pikepdf` (снимает restrictions через qpdf) или `PyMuPDF.authenticate(password)`. **Restrictions без encryption** (запрет печати/копирования) — не препятствуют программному чтению, `qpdf` снимает их легально. **Watermarks** не мешают парсингу.

**Правовая база:** ст. 1299 ГК РФ запрещает обход технических средств защиты, но **п. 4 той же статьи** допускает обход при lawful use, когда иначе пользоваться правомерно полученным произведением невозможно. Поскольку тендерная документация по 44-ФЗ **обязательна к публичному размещению**, парсинг публичных тендеров однозначно правомерен. Осторожнее с пользовательскими соглашениями коммерческих ЭТП — они могут запрещать автоматизацию; юридический риск переходит от ст. 1299 к договорному.

**JavaScript-защита от копирования на сайте** (отключение правой кнопки, DRM в просмотрщике) влияет только на ручное копирование через браузер — после скачивания файла парсер работает как обычно.

### Рекомендуемая архитектура pipeline

Поток обработки тендера выстраивается в семь стадий с чёткими отказами в сторону ручной модерации. **Ingestion** скачивает пакет с ЕИС (через Clearspending в MVP, позже — собственный SOAP-клиент с токеном), прогоняет антивирус и рекурсивно распаковывает архивы с защитой от zip-bomb и корректной декодировкой CP866/CP1251. **Document router** определяет MIME через `python-magic` и направляет файлы: XLSX → openpyxl-ветка, DOCX → python-docx, PDF → subrouter (текстовый слой через pdfplumber vs OCR через Яндекс Vision/PaddleOCR), IFC → IfcOpenShell, GGE → кастомный XSD-парсер, DWG → ODA File Converter + ezdxf для штампа или OCR-pipeline для визуалки.

**Layout detection** для PDF (Docling или DocLayNet + LayoutLMv3) сегментирует документы на ТЗ / проект контракта / смету / приложения — это принципиально для последующего выбора промпта. **Extraction tier** работает каскадом: Tier-1 на дешёвой модели (GPT-4o-mini, YandexGPT 5 Lite, GigaChat Lite) классифицирует и собирает грубые поля; Tier-2 на основной модели (GPT-4.1, Claude Sonnet 4.6, YandexGPT 5 Pro, GigaChat Max) извлекает TenderSummary через Instructor + Pydantic; Tier-3 reasoning-модель (o3, Opus 4.6, T-Pro 2.1 self-hosted) привлекается только при конфликтах в числовых полях и для CoT-валидации.

**Validation layer** — самая недооценённая часть. Pydantic-схема с валидаторами ИНН/КБК, regex-grounding всех чисел в исходном тексте, арифметическая сверка сумм смет (позиции ± 0,5% = НМЦК), rules engine по 44-ФЗ/223-ФЗ/ПП 2571/1875, cross-model consensus на критичных полях. Без этого слоя LLM будет давать 10–20% ошибок на числах.

**Storage:** PostgreSQL + pgvector для эмбеддингов + S3/MinIO для исходников + отдельный audit log (hash документа, версия модели, промпт, response) — для разбора «почему система извлекла такое число».

### Что делать сейчас vs что отложить

**В MVP включить:** XML извещений (через Clearspending), распаковку архивов с корректной кириллицей, pdfplumber+pymupdf4llm для цифровых PDF, openpyxl+calamine+rapidfuzz+LLM для XLSX-ВОР, кастомный парсер `.gge` по XSD, Tesseract rus + Яндекс Vision для сканов, Camelot lattice + Яндекс Vision table для таблиц, hybrid LLM-каскад с Instructor+Pydantic+source grounding+cross-model validation, опциональную ветку IFC через IfcOpenShell.

**Отложить на v2:** собственный SOAP-клиент ЕИС с ГОСТ-сертификатом и stunnel, геометрический расчёт площадей из DWG, fine-tuning T-Pro 2.1 на доменном корпусе, TIMML, полноценный RAG-слой с историей тендеров, автогенерация готовой (не прототипной) сметы с валидацией по ФСНБ-2022.

**Не делать никогда в закрытом продукте:** LibreDWG (GPLv3), Marker с базовой лицензией в коммерческом продукте, прямой парсинг RVT, полагание на Autodesk APS/Forge из РФ, Textract для критичных шифров ФЕР без перепроверки.

## Заключение: где проходит граница автоматизации

Автоматизировать **извлечение структурированных данных из тендерной документации реально** на уровне 85–95% точности при грамотной архитектуре — достаточно для автогенерации **прототипа сметы**, требующего человеческой верификации. Невозможно полностью автоматизировать: восстановление смысловой структуры из DWG, однозначную интерпретацию нестандартных XLSX без доменных правил, безошибочное извлечение чисел из низкокачественных сканов. **Главные инженерные вызовы лежат не в выборе модели, а в трёх слоях:** (1) корректная обработка кириллических имён в архивах, (2) доменная пост-обработка таблиц с валидацией сумм и справочником ФЕР/ГЭСН, (3) защита от LLM-галлюцинаций на числах через source grounding и cross-model consensus.

Главный парадокс экосистемы на апрель 2026: **самые ценные форматы (`.gge`, XML ЕИС, IFC) не имеют зрелых open-source парсеров**, тогда как наименее структурированные (PDF) получили за последние два года мощный арсенал инструментов (Docling, Marker, olmOCR, PaddleOCR-VL 1.5). Это создаёт окно возможностей: построить первый общедоступный Python-парсер `.gge` под конкретный продукт и превратить его в стратегический актив. Следующая волна изменений — массовое внедрение XML-ВОР после 11.01.2025 и рост доли IFC в тендерах — сделает детерминированные парсеры ещё ценнее, чем LLM-извлечение. Ставка в MVP на гибридную архитектуру с детерминированным ядром и LLM-страховкой не только экономически оптимальна (4–170 ₽/тендер вместо 300–500 ₽ на чистом vision-LLM), но и создаёт правильную траекторию развития продукта в горизонте 2–3 лет.
